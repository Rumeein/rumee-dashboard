Rumee — Product & Inventory Model

Where everything lives, before it's for sale

RESOLVED — shipped & live
2026-07-18

This is the model we agreed on, drawn out the first time as a question — now shipped. Everything you sell is still a Design + Variation in Product Master. Everything that goes into making it — bought or assembled — still passes through exactly one ledger first, Item Master. The part that used to be a gap, a BOM's Output now resolves to a real Product Master doc ID instead of a typed name, is built and live.

the hub — every material passes through here confirmed live, 2026-07-18
01

Vendor Purchase

You buy something — thread, a clasp, a box, or a finished earring from a job-worker. Doesn't matter what category it is.

02

Item Master (Materials)

Everything you buy lands here, in one ledger, no exceptions. A copper Bahubali chain, a packaging box, and a DJ-7 earring you bought from a vendor are the same kind of record — just tagged with a different type. There is no separate "stock" system: a Material's current_stock field is your stock. The Stock tab is a window onto this table, not a second place things go. A purchase line whose Internal Item Name doesn't match anything here now prompts to add it on the spot, instead of silently creating a record.

03

BOM — now declares where its output goes

A BOM is still a ratio: "1 unit of DJ-7 Bahubali needs 1 DJ-7 earring + 1 copper Bahubali chain." What changed is the Output field. Intermediate still runs through a Conversion Batch — consumes ingredient stock, produces stock of a new Material (the chain), which can itself be a component of something else later. Final skips that step: it points straight at a real Product Master doc, chosen from a live picker that refuses to save if nothing matches.

04

Product Master

Still the only place "what we sell" lives — one doc per Design + Variation (DJ-7 Bahubali, DJ-7 OG), holding the real Flipkart / Meesho / Amazon listings. It still knows nothing about cost or stock, by design — Jaiswal's own words, "BOM should not be filled into product master." Instead it now shows a BOM / No BOM chip per item, so progress across all ~85 items is visible at a glance.

✓

The gap — closed

This page used to end on an open question: should a BOM's output and its Product Master doc become the same record, or stay two records that reference each other by ID? Answer: linked-by-ID. A Final BOM's Output field is matched against Product Master's real doc IDs at save time — type a name that doesn't match anything real and it refuses to save. No more drift from a typo.

✓

Sale and return, live

An order shipping now resolves to its Product Master item, walks that item's Final BOM, and decrements each ingredient Material's stock directly in Item Master — the finished product itself was never stocked separately, so this is the only place stock moves for a sale. An intact-marked return or RTO credits the same components back, per-component (earring / box / chain), reusing the exact intact/damaged signal the Returns Scanner already captures for QC bonus payouts. A sale that can't be resolved to a Product Master doc surfaces on a "Needs Mapping" card and in Discord — never silently guessed.

Shipped, live, confirmed

Built as a 4-item sequence, one verified before the next, then confirmed by Jaiswal on the real deployed site — signed in, both via a local dev server and the live URL:

  • Purchases: Internal Item Name prompts to add a missing item instead of silently creating one.
  • BOM: Output Type field, Final path matched live against Product Master, refuses to save on no match.
  • Product Master: BOM / No BOM chip per item, plus an overall progress counter.
  • Pipeline: order ships → stock decrement; intact return/RTO → stock credit-back; unresolved SKUs surfaced, never guessed. firestore.rules published and confirmed matching in the Firebase Console.
